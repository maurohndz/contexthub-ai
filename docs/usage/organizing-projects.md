# Organizing Projects

How to structure knowledge across organizations and projects so answers
stay relevant.

## The two containers

```
Organization  → the tenant: members, AI credentials, and all knowledge
 └─ Project   → one initiative/system/team: its sources + its conversations
```

- **Organizations isolate everything.** Members, documents, credentials
  and conversations never cross organizations. You can belong to several
  and switch from the left sidebar (**Organizaciones**).
- **Projects scope retrieval.** When you ask a question, Ágora searches
  *only the active project's* sources. This is a feature: a billing
  question answered from the billing project cannot be polluted by an
  unrelated HR manual.

## Choosing project boundaries

Good project = a coherent body of knowledge someone would query as a
unit:

- ✅ *Payments platform* — its manuals, rules, minutes and decisions.
- ✅ *Client X implementation* — everything about that engagement.
- ❌ *All company documents* — retrieval degrades when everything
  competes with everything.
- ❌ One project per document — impact analysis and contradiction
  detection need related sources living together.

## Working day to day

- Create projects from **Nuevo proyecto** in the right sidebar; each row
  shows its document count and last activity, updating in real time.
- Each project keeps your **per-user chat history** underneath it when
  active — switch threads freely, context never leaks between projects.
- Manage a project's sources from its menu (⋮ → **Ver fuentes**).

## Multi-user today, roles tomorrow

Today, organization members share the organization's projects and
knowledge, while conversations remain private per user. A full **roles &
permissions model** (who can upload vs. only ask; org-level vs
space-level rights) plus email invitations is the core of
[roadmap Phase 2](../roadmap.md).
